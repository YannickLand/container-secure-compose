# v1.0.0 — Initial release

A privilege-minimised Docker Compose generator that produces security-aware
`docker-compose.yml` files from an abstract application description — without
requiring developers to know the details of Docker security options.

## What is this?

`container-secure-compose` separates *what a service needs to do* from *how
Docker should be configured to do it securely*. Developers name their services
and pick composable building blocks; the tool resolves the blocks into a
complete, hardened `docker-compose.yml` and prints a security report showing
the privilege level of every service. Insecure escalations are explicit
opt-ins — visible in the input config, not buried in generated output.

## Features

### Core functionality
- **`csc generate`** — reads an `app_config.yaml`, resolves building blocks, writes a hardened `docker-compose.yml`, and prints a security report
- **`csc validate`** — validates an `app_config.yaml` without generating any output
- **`csc list-blocks`** — lists all available building blocks with their security impact level and escalation flag
- **`csc explain`** — prints the merged configuration and security metadata for a single block
- **`--stdout`** flag — prints generated YAML to stdout instead of writing a file
- **`--no-report`** / **`--report-format json`** — control security report output

### Building blocks
- **`standard`** — secure default for application containers: `cap_drop: ALL`, `no-new-privileges:true`, `user: nobody`
- **`non-root`** / **`root`** / **`drop-privileges`** / **`drop-capabilities`** — user and capability presets
- **`read-only`** — read-only root filesystem
- **`init`** / **`wait-init`** — PID 1 init process and startup ordering
- **`sidecar`** — sidecar service template
- **`privileged`** / **`host-network`** / **`add-cap-net-admin`** / **`add-cap-net-raw`** — named escalation blocks; choosing them makes the risk visible by name
- **`layer2-network`** / **`mirroring-setup`** / **`no-new-privileges`** — specialised network and security options

### Configuration
- Declarative YAML input with full Pydantic validation before any files are written
- Per-service `properties` map for arbitrary Docker Compose keys not covered by a block
- Auto-detected `building_blocks/` directory next to the config file or in CWD
- Warning on deprecated `version:` key in config

### Security reporter
- Per-service impact assessment (`low` / `medium` / `high` / `critical`) derived from the blocks chosen
- Escalation flag highlights services that opt in to elevated privileges
- Text and JSON output formats; suppressible with `--no-report`
- `csc diff` — compares a generated compose to an existing file and reports security regressions and improvements

### Infrastructure
- Pure-Python package installable with `pip install -e .`
- `pyproject.toml`-based build (setuptools), entry point `csc`
- Python 3.11+ (type-annotated, Pydantic v2 models)

### CI / quality
- GitHub Actions pipeline: ruff (lint), bandit (source scan), pip-audit (dependency scan), mypy (type check), pytest with coverage, Codecov upload
- 110 unit tests, 99% line coverage
- Integration test gaps documented in `tests/` (`conftest.py` fixtures cover real building blocks)

### Documentation
- Full [Diataxis](https://diataxis.fr) documentation structure under `docs/`: tutorial, three how-to guides, explanation pages, and CLI + configuration reference
- Getting-started tutorial walks through installing and generating a compose file for the bundled `ping-tracker` example

## Requirements

- Python 3.11 or newer
- PyYAML, Click, Pydantic v2 (installed automatically via `pip install`)
- Docker Engine (to run the generated compose files)

## Quickstart

```bash
pip install -e .
csc generate examples/ping-tracker/app_config.yaml
```

See [docs/tutorials/getting-started.md](docs/tutorials/getting-started.md) for a full walkthrough.
