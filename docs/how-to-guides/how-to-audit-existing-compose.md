# How to audit an existing docker-compose.yml

## When to use this

Use the `csc audit` command when you have an existing `docker-compose.yml`
that was **not** generated by container-secure-compose and you want to
understand its security posture — for example, a file you inherited, a
third-party example, or a project that predates this tool.

`csc audit` does not require a CSC config file. It reads any Compose YAML
and produces the same security report that `csc generate` prints.

---

## Steps

### 1 — Locate the Compose file

Find the `docker-compose.yml` (or `compose.yaml`) you want to audit:

```bash
# Example: a file in the current directory
csc audit docker-compose.yml
```

### 2 — Read the security report

The report shows one row per service:

```
Auditing: docker-compose.yml

Security report
---------------
  Service         cap_drop   no-new-priv  non-root   read-only  impact    notes
  -------------------------------------------------------------------------
  web             no         no           no         no         medium
  worker          yes        yes          yes        no         low
```

Each column:

| Column | Meaning |
|---|---|
| `cap_drop` | `cap_drop: ALL` is present |
| `no-new-priv` | `security_opt: no-new-privileges:true` is present |
| `non-root` | `user` is set to a non-root identity |
| `read-only` | `read_only: true` is set |
| `impact` | Derived worst-case label: `low / medium / HIGH / CRITICAL` |
| `notes` | Active escalations (host-network, cap_add, privileged) |

### 3 — Get JSON output for automation

Pass `--report-format json` to pipe the report into another tool:

```bash
csc audit docker-compose.yml --report-format json | jq '.[] | select(.impact != "low")'
```

---

## Fixing findings

After identifying the issues, you have two options:

**Option A — Edit the Compose file directly**

Add the missing security options to each service. See the
[Building blocks reference](../reference/building-blocks-reference.md) for the
exact YAML snippets each building block provides.

**Option B — Migrate to a CSC config**

Write an `app_config.yaml` (see the [Getting started tutorial](../tutorials/getting-started.md))
and use `csc generate` to produce a hardened Compose file going forward. Use
`csc diff` to verify the migration does not introduce regressions.

---

## PowerShell syntax note

On Windows PowerShell, quote the `--report-format` value:

```powershell
csc audit docker-compose.yml --report-format json
```
